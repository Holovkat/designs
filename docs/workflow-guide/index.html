Overview

One workflow, many project shapes.

This guide describes a generic operating model for moving work from idea to production with clear planning, approval, implementation, QA, release, and closeout gates.

The README is the repository index: it points readers to the command pack, templates, skills, and rendered guide. This HTML page is the detailed canonical guidance for how the workflow should be run inside a project.

Document contract

Audience
Project owners, planning leads, builder agents, reviewers, QA participants, and release approvers.
Scope
Reusable delivery workflow guidance that can be applied to different stacks and hosting models.
Authority
Project instructions and accepted issue scope always decide local policy where they are stricter.
Evidence
Every stage should produce traceable notes, commands, links, test results, or approval records.
Installation

Install the command and skill pack where the harness can discover it.

The workflow pack can be installed globally for a harness or locally inside a project. Project-local setup is usually safer when the repository has its own release authority, QA environment, approval model, hooks, scripts, or worktree policy.

Installed surface

  • Command markdown for planning, plan review, build, closeout, DoD review, compliance, and UAT.
  • Project skills for worktree lifecycle, plan review, and DoD review.
  • Optional hooks, scripts, worktree guidance, settings, and design templates.

Planning Intake

Start with the right planning command.

Planning commands turn a request into a reviewed packet before implementation begins. Choose the command based on the source and shape of the work, not on the tool preference of the current agent.

Entrypoints

/plan-feature
Use for new capabilities, workflow changes, interface additions, or multi-step improvements.
/plan-bugfix
Use when observed behavior differs from expected behavior and regression coverage is needed.
/plan-github
Use when an existing issue or pull request needs to be converted into the standard hierarchy.

Intake rule

Do not treat a planning command as permission to build. Intake creates a candidate scope, evidence requirements, and acceptance criteria. Implementation waits for explicit planning approval.

If the request mixes unrelated outcomes, split it before planning. If requirements are missing, record questions and recommended options rather than filling gaps with assumptions.

Planning Decomposition

Convert intent into a traceable work hierarchy.

Planning decomposes work into an epic or parent issue, phase or sprint issue, task issues, and a local checklist mirror that tracks issue numbers, signoff state, and required evidence.

Hierarchy

Epic or parent
Defines the business outcome, non-goals, primary risks, and completion contract.
Phase or sprint
Groups related work into an approval-friendly delivery slice with visible dependencies.
Task issues
Carry acceptance criteria, implementation boundaries, tests, and evidence expectations.
Checklist mirror
Records issue numbers, status, blockers, and signoff without replacing the issue source of truth.

Planning packet

A planning packet is ready for approval when scope, acceptance criteria, implementation slices, review gates, QA scenarios, and known risks can be read without relying on chat memory.

Review Gates

Review the plan before build and the build before handover.

Two review gates close the gap between planning intent and delivered work. /plan-review compares the planned artifacts against the Q&A or intake record before build approval. /dod-review compares approved requirements against delivered build outcomes before user handover or UAT.

Plan vs Q&A review

Use this gate after the plan is drafted but before implementation starts. It checks that accepted requirements, non-goals, risks, dependencies, acceptance criteria, and evidence gates from the Q&A session or imported issue are represented in the epic, sprint, task issues, and checklist mirror.

If the plan misses a decision or invents scope, return to the planning command before any build session starts.

Requirements vs build review

Use this gate after implementation and local validation, but before handover to the user, QA, or UAT. It maps each accepted requirement to the delivered files, behavior, tests, review findings, compliance evidence, smoke results, and known caveats.

Passing automated checks alone is not Definition of Done. The rank depends on requirements coverage and evidence quality.

Build Session

Start implementation with a frozen packet.

/start-session begins governed implementation. It should confirm the approved task packet, branch or worktree policy, local runtime expectations, and the files or modules owned by the task.

Session rules

  • Use isolated workspaces, worktrees, environments, or runtimes when the project requires them.
  • Keep implementation scoped to the approved task packet and declared ownership boundary.
  • Do not mix unrelated cleanup, formatting, or refactors into the same slice.
  • Record commands and evidence needed by reviewers, testers, and later closeout.

Ownership snapshot

Branch policy
Use the current project branch or create the required branch/worktree only when policy says to.
Task packet
Freeze acceptance criteria, owned files, non-goals, and validation before edits start.
Changes
Prefer small, atomic commits or handoff slices that can be reviewed without unrelated context.
Blockers
Stop for unclear authority, mixed environments, missing credentials, or approval gaps.
Builder Gates

Separate writing from evaluation.

Builder sessions work best when writer roles and read-only gates stay distinct. A writer can propose fixes and run local validation, but review, test, and compliance gates should remain independently inspectable.

Correction loop

When a read-only gate finds an issue, return the finding to the writer with evidence and expected remediation. The writer fixes within the approved packet, reruns relevant checks, and sends the same gate back through review.

CI/CD Authority

The project defines the release path.

CI/CD authority is provider-neutral. A project may define GitHub Actions, Vercel, another hosted platform, a private pipeline, or a manual approval workflow as the authority. Those are examples, not universal assumptions.

Discovery checklist

  • Read project instructions, repository docs, workflow files, and hosting configuration.
  • Identify the canonical branch, deployment target, environment name, and domain mapping.
  • Confirm which actor or workflow is allowed to deploy, promote, remove, or clean up resources.
  • Attach evidence to the governed path, not to arbitrary previews or local-only state.

Hard rule

Agents must not manually deploy, redeploy, promote, remove, or clean up hosted resources unless the user explicitly asks for that exact action. If authority is unclear, stop and ask which environment and workflow are canonical.

QA and UAT

Acceptance evidence belongs to the canonical QA environment.

QA readiness starts after the governed integration path has produced a reviewable build. Evidence should identify the canonical QA URL or environment, deployed identity, scenarios, accounts or roles, known limits, and the path that produced the build.

Readiness manifest

Environment
Canonical QA URL or environment identifier, plus any required access constraints.
Build identity
Commit, artifact, deployment, or release candidate identifier from the governed path.
Scenarios
Approved UAT cases, expected outcomes, setup data, and evidence capture instructions.
Known limits
Accepted caveats, blocked areas, data constraints, or follow-up tasks.

/uat outcome

/uat validates approved scenarios and records evidence. It should produce one of two outcomes: rework with concrete failures and reproduction steps, or approval with signoff evidence tied to the canonical QA environment.

Production

Promotion requires explicit owner approval.

Production work follows the governed production path only after QA/UAT approval and explicit owner approval. Do not infer approval from passing checks, preview availability, or a successful QA smoke.

Promotion packet

  • Owner approval with the production target and release scope named.
  • Governed promotion path and actor responsible for the production action.
  • Rollback or mitigation plan, including where evidence will be recorded.
  • Production smoke steps and final signoff expectations.

Evidence standard

Production evidence must come from the user-accessible production environment and the approved deployment path. Random preview evidence, stale aliases, unknown build identities, or local-only checks do not prove production readiness.

Closeout

Finish the work, not just the code.

Closeout confirms that the project can understand what changed, why it changed, how it was verified, and what remains. It is the bridge between delivery evidence and future maintainability.

Closeout checklist

  • Run required quality gates and record command results.
  • Update project docs, release notes, or checklist state when required by the project.
  • Retain evidence for planning, implementation, QA/UAT, production, and final signoff.
  • Clean temporary files, branches, worktrees, or environments only under project policy.
  • Record lessons learned when a reusable issue, risk, or workflow gap was discovered.

Handoff note

A good final handoff names changed files, commands run, verification results, residual risks, and any follow-up tasks. It should be short enough to scan and complete enough to audit later.

Overlays

Project-specific gates can extend the base model.

The workflow is generic by default, but projects may add stricter overlays for domains where failure risk, audit needs, or operational complexity is higher.

Common overlay categories

Data
Backups, migration plans, readback checks, retention, restore testing, and data ownership.
Security
Access review, secret handling, permission boundaries, threat notes, and audit evidence.
Regulated flows
Approval records, traceability, separation of duties, and retained validation artifacts.
Infrastructure
Change windows, capacity checks, rollback commands, monitoring, and owner notification.
Tenants
Environment separation, tenant-safe test data, role boundaries, and customer-specific signoff.
Performance
Load expectations, latency budgets, regression baselines, and post-release monitoring.

Overlay rule

Overlays can add gates, evidence, and stop conditions. They should not bypass the base workflow, weaken deployment authority, or replace explicit QA and production approvals.

Keep overlay examples generic in shared guidance. Put project names, environment URLs, service names, tenant identifiers, and domain-specific runbooks in project-local documentation.
