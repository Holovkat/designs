Install Model

Install once globally or inside each project.

The command and skill pack is designed for two operating modes: a global harness setup that makes the workflow available everywhere, or a project-local setup that keeps the commands, skills, hooks, scripts, and worktree guidance inside one repository.

Prefer project-local installs when a repository has its own release authority, approvals, QA target, worktree policy, or domain-specific overlays. Use global installs for reusable baseline commands and skills that the harness can apply across many projects.

Source of truth

The template package lives under templates/instructional-documents/. The installer copies command markdown, shell scripts, hooks, project skills, and worktree guidance from that package into the target project.

Project Local

Install into one repository when project policy matters.

Run the project-local installer from the target repository root when commands, hooks, skills, scripts, and worktree guidance should travel with that project. 

cd /path/to/project
/path/to/designs/templates/instructional-documents/install-session-workflows.sh --dry-run
/path/to/designs/templates/instructional-documents/install-session-workflows.sh

Defaults are commands/, hooks/, and settings.json. Set WORKFLOW_COMMANDS_DIR, WORKFLOW_HOOKS_DIR, or WORKFLOW_SETTINGS_FILE when the harness expects a different project-local layout.

Common options

--dry-run
Preview copied files before changing the project.
--commands-only
Refresh commands, scripts, skills, and worktree guidance without hooks.
--hooks-only
Refresh hooks without changing commands or skills.
--with-templates
Copy design templates as well as the workflow pack.
--no-settings
Keep the project's existing harness settings file.
Global Harness

Install globally when the harness owns discovery.

Global installation is harness-specific. Copy the same command and skill sources into the harness directories that it reads at startup, then restart or reload the harness so it discovers the new files. 

# Example shape only. Replace destinations with your harness paths.
cp templates/instructional-documents/commands/*.md /path/to/harness/commands/
cp -R templates/instructional-documents/skills/. /path/to/harness/skills/
cp templates/instructional-documents/hooks/*.sh /path/to/harness/hooks/
chmod +x /path/to/harness/hooks/*.sh

Global install rules

  • Keep project-specific environment names, URLs, credentials, and release rules out of global files.
  • Let project-local instructions override global defaults when they are stricter.
  • Record the global source commit or package version so teams can reproduce the setup.
  • Run a command discovery check after reload so missing files are caught immediately.
Reusable Prompt

Common setup prompt for any harness

Use this prompt when asking an agent or automation harness to install the workflow pack. Fill the placeholders first so it can choose global or project-local setup without guessing. 

You are installing the Designs workflow commands and skills into an agent harness.

Source repository:
- Designs repo path or URL: [DESIGNS_REPO]
- Source package path: templates/instructional-documents/

Target mode:
- Choose one: global harness install / project-local install
- Target project root, if project-local: [PROJECT_ROOT]
- Project-local command directory, if project-local: [PROJECT_COMMANDS_DIR or PROJECT_ROOT/commands]
- Project-local hook directory, if project-local: [PROJECT_HOOKS_DIR or PROJECT_ROOT/hooks]
- Project-local settings file, if project-local: [PROJECT_SETTINGS_FILE or PROJECT_ROOT/settings.json]
- Harness command directory, if global: [HARNESS_COMMANDS_DIR]
- Harness skills directory, if global: [HARNESS_SKILLS_DIR]
- Harness hooks/settings directory, if applicable: [HARNESS_HOOKS_OR_SETTINGS_DIR]

Install requirements:
1. Inspect the source package before copying files.
2. Install commands from templates/instructional-documents/commands/.
3. Install skills from templates/instructional-documents/skills/.

6. Do not deploy, promote, remove, or clean up hosted environments during installation.
7. Verify command and skill discovery after installation.
8. Report changed files, installed commands, installed skills, skipped items, and any manual follow-up.

Verification

Prove the harness can see the pack.

Installation is not complete until the target harness can list or invoke the commands and skills from the intended scope.

Verification checklist

  • Command list includes the planning, review, release assessment, build, closeout, compliance/DoD, and UAT commands.
  • Skill discovery includes lifecycle, plan-review, release-assess, and vibe-fix skills.
  • Project-local installs keep existing project settings unless replacement was requested.
  • Hooks are executable when the harness uses hook scripts.
  • The target project records any local overlays or exceptions after installation.

Stop condition

If multiple harnesses, command directories, or project roots are possible, stop and ask which target is canonical before copying files. A partial install into the wrong scope is harder to audit than no install.
